# 渲染层与页面交互规则

## 当前前端形态
- 渲染层位于 `src/renderer`，当前主要是单文件 HTML 页面配合原生脚本与 Vue 3 CDN 运行。
- 修改页面时，先阅读同文件现有组织方式，保持模板、状态、事件和样式写法一致。
- 不要因为局部需求直接引入新的前端框架、打包器或组件体系，除非用户明确要求。

## Vue 与状态组织
- 页面交互优先复用现有 `Vue` 响应式状态、计算属性、监听器和方法组织方式。
- 简单交互直接沿用当前页面内实现；不要为了小改动拆成新的复杂前端架构。
- 如果某段逻辑强依赖主进程数据，优先通过 preload API 驱动，而不是在页面层缓存第二份事实来源。
- 涉及导航当前页、激活会话、轮询范围等跨层运行态时，要明确单一事实来源；如果渲染层保留镜像状态，初始化和异常回退时必须主动与主进程同步。

## 页面能力边界
- 渲染层负责展示、输入、局部交互和调用 preload API，不负责直接访问 Node.js、store 或敏感认证信息。
- 新增设置项、列表操作或状态展示时，要同步确认 preload 和主进程是否已有对应接口；没有时按现有模式补齐链路。
- 页面上的文案、状态和列表结构应以当前业务流程为准，不默认沿用历史 BrowserView 嵌入式聊天假设。
- 新增业务页时，优先用独立脚本、独立页面容器或独立文件承载，再回到 `index.html` 做最小挂载。
- `index.html` 更适合作为导航、容器和视图切换层，不适合作为多个新模块的完整实现承载点。
- 当同一业务模块同时包含结构、样式和交互时，优先拆成独立 `.js` / `.css` 文件，再通过共享入口按顺序挂载。
- 渐进式模块化时，优先抽离完整业务域或子面板，不要只把零散函数搬文件后继续依赖大块全局状态。

## 样式与交互
- 优先复用当前页面已有的 DOM 结构和样式命名，不无故重置整页视觉层次。
- 小范围样式调整尽量局部修改，避免影响整页布局或多面板联动。
- 涉及消息列表、店铺切换、设置表单、调试面板等区域时，要验证交互状态与主进程回传数据一致。
- 视图加载函数、列表渲染函数和详情渲染函数不要把异常直接冒泡到导航切换层；页面专属异常应在当前页面内兜底为错误提示、空态或日志，避免左侧选中态回弹。
- 对 `getElementById`、`querySelector`、IPC 返回结果和接口回退数据的直接访问，要先做空值与结构校验；渲染层允许显示失败提示，不允许因为单个节点或单条脏数据让整页切换失败。

## 验证要求
- 修改渲染层后，至少验证主要视图可打开、核心按钮可操作、关键状态能正确刷新。
- 如果改动涉及 IPC 或 preload 数据回流，要同时检查页面展示与主进程行为是否一致。
- 涉及左侧导航、BrowserView 显隐或接口页初始化时，额外验证“切换成功、切换失败、窗口重载后恢复”三种场景，确认不会出现左侧 tab 与右侧内容不一致。
