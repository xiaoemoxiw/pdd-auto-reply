# 渲染层模块拆分与入口收口

当任务涉及 `src/renderer/index.html` 过大、多人容易冲突、页面私有逻辑需要迁出独立文件，或拆分后怀疑仍有旧逻辑残留时，使用这个技能。

## 目标

- 保持 `index.html` 只承担导航、容器、初始化和少量共享接线
- 优先按完整业务域拆分，不做零散函数搬家
- 让新模块可独立维护，同时不破坏现有页面联动

## 适用场景

- 需要把某个完整页面或面板从 `index.html` 迁到独立 `.js` / `.css`
- 需要设计 `window.__xxxModuleAccess` 这类共享 runtime bridge
- 需要排查“新模块已接入，但旧逻辑还在 index.html 同时生效”
- 需要评估某块逻辑该留共享接线层还是应继续迁出

## 默认原则

- 先遵循 `renderer-ui.mdc`、`pdd-core.mdc` 和共享入口协作规则
- 优先拆完整业务域，例如 `chat-api`、`shops`、`qa`、`quick-phrases`
- 页面私有状态、渲染函数、事件绑定优先放进模块
- `currentView`、活跃店铺、共享短语、全局 modal、日志等跨页面事实优先保留单一来源
- bridge 只做共享能力注入，不承接整块业务实现

## 工作流程

1. 先识别这块逻辑是完整业务域、共享能力，还是顶层接线
2. 列出状态、函数、事件、IPC 回流和跨模块依赖
3. 把页面私有渲染与事件优先迁出，再决定是否继续迁状态编排
4. 若需要共享事实，先在入口层收口成 `window.__xxxModuleAccess`
5. 删除 `index.html` 中重复旧逻辑，避免双绑定和同名函数覆盖
6. 验证模块挂载、跨模块联动和启动是否正常

## 保留在入口层的典型内容

- 导航切换与页面激活
- 顶层生命周期与初始化
- 全局 modal、日志、转义等通用工具
- 跨页面共享事实来源
- 少量必须统一编排的刷新或恢复逻辑

## 优先迁出的典型内容

- 页面私有状态
- 列表渲染、详情渲染、面板渲染
- 页面内事件绑定
- 局部按钮控制器
- 页面专属 IPC 回流处理

## 验证清单

- 新模块脚本是否已接入并成功初始化
- `index.html` 是否还残留同名函数或旧事件绑定
- bridge 是否只暴露共享事实，没有重新塞入业务实现
- 页面打开、关键按钮、跨模块刷新是否正常
- `GetDiagnostics`、`node --check`、`npm start` 是否通过

## 避免

- 只搬几个函数，剩余大块状态和事件仍留在入口层
- 新旧实现并存，导致重复绑定或互相覆盖
- 为了拆分制造新的全局状态中心
- 把真正的共享接线误拆成业务模块，破坏跨页面一致性
