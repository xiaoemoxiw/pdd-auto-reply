# 自动回复规则实现与调优

当任务涉及关键词规则、匹配评分、默认兜底、未匹配记录、话术库或 AI 意图识别时，使用这个技能。

## 目标

- 保持规则结构、主进程逻辑和页面配置能力一致
- 先定位问题在规则数据、匹配算法、兜底流程还是页面保存链路
- 用可复现的最小输入验证命中结果，而不是凭感觉调规则

## 默认要求

- 先遵循 `reply-engine.mdc`、`main-process.mdc` 和 `renderer-ui.mdc` 的实现边界
- 规则逻辑优先复用 `ReplyEngine`、store 默认配置、preload API 和现有页面编辑流
- 涉及 AI 意图时，继续复用 `ai-intent.js`、`system-phrases.js` 和 `aiIntent` 存储结构
- 修改前先确认当前问题是规则未保存、命中不准、兜底异常还是 UI 展示错误

## 任务分类

开始前先判断当前任务属于哪类：

- 新增或调整关键词规则
- 匹配评分、优先级或排除词调优
- 兜底话术、场景分类或冷却策略调整
- 未匹配消息记录与调试链路排查
- AI 意图模型、阈值、来源或命中结果调整

## 工作流程

1. 先确认相关配置来自主进程 store 还是页面临时状态
2. 检查规则结构是否与 `ReplyEngine` 的真实字段保持一致
3. 用最小示例验证 `match`、`testMatch` 或 `matchWithFallback` 的结果
4. 如果问题涉及页面编辑，再检查 preload API 和保存回写链路
5. 如果问题涉及 AI 意图，再确认模型状态、阈值、语料与回退策略
6. 交付时说明命中逻辑、兼容处理和验证结果

## 规则排查

- 优先确认 `keywordGroups`、`keywords`、`excludeKeywords`、`matchType`、`priority`、`minScore` 是否匹配当前需求
- 修改评分逻辑时，要同时检查普通命中、测试命中和兜底命中三条路径
- 正则规则必须保留容错，避免用户输入或错误配置直接中断流程
- 如果新增字段，要同步考虑 store 默认值、保存接口和页面编辑能力

## 兜底与记录

- 所有规则不命中时，先看场景分类，再看通用兜底文本
- 调整 `delay`、`cooldown`、`strategy`、`cancelOnHumanReply` 时，要考虑人工介入和重复触发
- 与未匹配消息相关的改动，要确认 `unmatchedLog` 仍完整记录并可在页面查看

## AI 意图排查

- 先确认模型是否已下载或加载，再判断问题在意图语料、相似度阈值还是回退逻辑
- 模型来源、缓存目录、本地导入和下载进度应继续沿用现有配置与状态字段
- 不要假设 AI 命中一定优于关键词规则；需要明确当前链路中的优先级关系

## 输出要求

完成后需要说明：

- 改动或排查的是哪条自动回复链路
- 复用了哪些现有配置、模块或页面入口
- 如何验证命中、误判、兜底或 AI 状态
- 是否存在旧数据兼容或环境前置条件

## 避免

- 只改页面文案，不改真实保存和匹配逻辑
- 用硬编码话术或临时分支掩盖规则问题
- 让 AI 配置、关键词规则和兜底策略彼此脱节
