# 拼多多 API 与会话链路排查

当任务涉及店铺认证、会话列表、消息历史、发送消息、轮询同步、网络抓包或相关 UI 状态异常时，使用这个技能。

## 目标

- 先判断问题在认证、请求、解析、轮询还是页面展示
- 优先复用现有店铺管理和 API 客户端，不平行新建链路
- 用最小验证确认问题位置，再决定修改范围

## 默认要求

- 先遵循 `main-process.md` 和 `pdd-core.md` 的边界，再执行这里的流程
- 店铺认证优先沿用 `ShopManager`、`PddApiClient`、分店铺 partition 和当前 Token 导入方式
- 不在输出、日志、截图或文档中泄露真实 Token、Cookie、mallId 等敏感信息
- 分析时明确区分主进程状态、preload 暴露能力和渲染层展示状态

## 任务分类

开始前先判断当前任务更偏向哪一类：

- Token 导入或认证初始化失败
- 会话列表或消息历史拉取异常
- 发送消息失败或回执异常
- 轮询不同步、重复触发或事件不更新
- 抓包记录、日志或调试面板排查

## 排查流程

1. 先确认问题属于单店铺还是多店铺场景
2. 再确认是否与 `ShopManager` 的店铺信息、tokenInfo、partition 或活动店铺切换有关
3. 查看 `PddApiClient` 当前请求路径、请求头、认证初始化和返回解析逻辑
4. 判断失败点在 `_request`、`initSession`、列表解析、消息解析还是轮询调度
5. 如果 UI 有异常，再向上检查 preload API 和页面事件订阅是否一致
6. 交付时说明问题根因、实际改动点和验证方式

## 认证问题排查

- 先确认 Token 导入链路走的是当前有效入口，而不是旧 BrowserView 登录假设
- 如果会话初始化依赖隐藏窗口认证，优先检查 `initSession` 逻辑和当前页面跳转结果
- 如果响应提示认证过期，优先沿用现有 `authExpired` 事件与重试思路
- 涉及 Cookie、Referer、Origin、User-Agent、`X-PDD-Token` 时，以当前实现为准，不手写另一套约定

## API 与解析排查

- 优先复用 `_request`、`_get`、`_post` 以及当前返回解析函数
- 不要假设接口字段固定；延续现有多候选字段兼容方式
- 先确认错误发生在 HTTP 层还是业务返回层，再决定改请求还是改解析
- 如果问题只出现在特定接口，优先对照现有日志或抓包定位最小差异

## 轮询与事件排查

- 涉及消息同步时，同时检查轮询频率、去重缓存、事件派发和页面监听
- 先确认数据是否已经到达主进程，再判断是 preload 派发丢失还是页面状态未刷新
- 如果出现重复消息或重复自动回复，优先检查消息去重和多来源事件合流

## 输出要求

完成后需要说明：

- 问题位于认证、请求、解析、轮询还是 UI 回流
- 复用了哪些现有模块与入口
- 关键验证点是什么
- 是否还存在环境前置条件或待继续观察的问题

## 避免

- 把历史 BrowserView 注入式方案当成当前默认链路
- 直接打印真实凭据做排查
- 为局部问题再造一套 API 客户端、事件总线或状态管理
