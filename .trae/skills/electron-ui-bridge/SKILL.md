# Electron 页面联动与桥接实现

当任务横跨 `src/renderer`、`src/preload` 和 `src/main`，需要新增页面能力、打通 IPC 链路或排查状态不同步问题时，使用这个技能。

## 目标

- 先确认展示层、桥接层和主进程职责边界
- 按现有 Electron 模式完成联动，不把业务逻辑塞回页面层
- 让新增能力在页面可用、在主进程可维护、在 preload 可控

## 默认要求

- 先遵循 `pdd-core.md`、`main-process.md` 和 `renderer-ui.md`
- 页面层只负责交互与展示，主进程负责真实状态与副作用
- preload 继续作为唯一桥接入口，不在页面中直接访问 Node.js 能力
- 修改前先确认当前页面是否已经有相近 API、事件订阅或状态组织方式

## 任务分类

开始前先判断需求更偏向哪类：

- 新增页面按钮、表单或面板能力
- 新增或调整 preload API
- 新增或调整主进程 IPC
- 页面状态不刷新、事件不回流或展示与真实状态不一致

## 工作流程

1. 先明确能力属于读取、写入、事件订阅还是长流程操作
2. 从页面需求反推 preload 接口，再反推主进程 IPC 和真实状态来源
3. 优先复用现有 `window.pddApi` 风格、命名和数据结构
4. 让主进程承担数据校验、持久化和副作用，页面只消费结果
5. 如涉及事件回推，确认主进程发事件、preload 订阅包装和页面监听三段都通
6. 交付时说明页面入口、桥接接口、主进程改动点和验证结果

## IPC 设计要求

- 新增接口优先沿用 `invoke` 和 `onXxx` 的现有组织方式
- 返回值尽量保持稳定、明确，避免把主进程内部对象直接透给页面
- 如果一个能力已有相近接口，优先扩展现有命名族，不平行再起一套

## 页面实现要求

- 优先复用现有 Vue 状态、方法、监听器与 DOM 组织方式
- 局部需求尽量局部修改，不为小交互引入新的页面架构
- 状态来源要单一，页面不要缓存与主进程冲突的第二份事实

## 验证清单

- 页面是否能正常触发 preload API
- 主进程是否真正执行了对应逻辑
- 数据回流后页面是否刷新到正确状态
- 是否影响已有按钮、事件监听或其它面板行为

## 避免

- 页面直接访问 Node.js 或 store
- 为了赶进度跳过 preload，直接让渲染层耦合主进程实现
- 页面、preload、主进程三层各维护一份不一致状态
